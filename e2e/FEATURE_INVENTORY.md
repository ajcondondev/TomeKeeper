# TomeKeeper — Feature Inventory

> Phase 1 analysis. Generated by exploring the live app with Playwright MCP and reading source code.
> App: http://localhost:5174 · API: http://localhost:3001

---

## 1. Pages & Routes

| Feature | Route | Auth Required | Risk Priority | Complexity |
|---------|-------|--------------|--------------|------------|
| Login | `/login` | No (public) | **Critical** | Low |
| Registration | `/register` | No (public) | **Critical** | Low |
| My Library | `/library` | Yes | **Critical** | High |
| Reading List | `/reading-list` | Yes | **High** | Medium |
| My Reviews | `/reviews` | Yes | **High** | High |
| 404 Not Found | `/*` (catch-all) | Yes (inside AppShell) | Low | Low |
| Root redirect | `/` | Yes | Low | Low |

**Notes:**
- Unauthenticated visits to protected routes (`/library`, `/reading-list`, `/reviews`) redirect to `/login`.
- Authenticated visits to `/login` or `/register` redirect to `/library`.
- 404 is rendered inside the AppShell (sidebar + navbar visible) for authenticated unknown routes.

## 3. Critical End-to-End User Workflows

### WF-1: Registration → Library
1. Visit `/register`
2. Enter valid email + password (≥8 chars) → click "Create account"
3. Auto-login → redirect to `/library`
4. Library is empty (empty state shown)

### WF-2: Login → Library
1. Visit `/login` (or any protected route when logged out)
2. Enter registered email + password → click "Sign in"
3. Redirect to `/library`

### WF-3: Sign Out → Login Required
1. Click "Sign out" in sidebar
2. Redirect to `/login`
3. Accessing `/library` now redirects back to `/login`

### WF-4: Add Book → Appears in Library
1. In `/library`, click "Add Book" → modal opens
2. Fill Title* + Author* (+ optional Genre, Pages, Cover URL)
3. Click "Add Book" → modal closes
4. Book card appears in library grid; book count increments

### WF-5: Change Book Status (Read / Want to Read)
1. From Library or Reading List, locate a BookCard
2. Click "Read" → book status changes to `read`, button label toggles to "Unread"
3. Click "Unread" → status reverts to `unread`
4. Click "Want to Read" → status = `want-to-read`, button toggles to "Remove"
5. Book now appears in `/reading-list`
6. Click "Remove" → status reverts to `unread`, book disappears from reading list

### WF-6: Delete Book
1. Click "Delete book" (trash icon) on a BookCard
2. Book is removed immediately; count decrements

### WF-7: Reading List (Derived View)
1. Navigate to `/reading-list`
2. Only books with status `want-to-read` are shown
3. Same BookCard actions available (Read, Remove, Delete)
4. Empty state shows if no `want-to-read` books — CTA links to `/library`

### WF-8: Add Review
1. In `/reviews`, click "Add Review" → modal opens
2. Select book from dropdown (all user's books listed)
3. Fill Review Title* + Review* textarea
4. Click "Add Review" → modal closes
5. ReviewCard appears; review count increments

### WF-9: Edit Review
1. On a ReviewCard, click "Edit" → Edit Review modal opens
2. Book field is read-only; Title and Review fields are pre-populated
3. Modify Title and/or Review → click "Save Changes"
4. ReviewCard updates with new content

### WF-10: Delete Review
1. On a ReviewCard, click "Delete"
2. Review is removed immediately; count decrements

### WF-11: Find Book Cover (External API)
1. In AddBookModal, enter a title (and optionally author)
2. Click "Find Cover" (enabled only when title is non-empty)
3. Cover URL field auto-populated from Open Library API
4. If no cover found: error "No cover found for this title."
5. If fetch fails: error "Could not reach Open Library. Check your connection."

## 4. Forms: Equivalence Partitions & Boundary Values

### 4a. Registration Form (`/register`)

| Field | Type | Required | Constraint | Valid Partition | Invalid Partitions |
|-------|------|----------|-----------|----------------|-------------------|
| Email | `type="email"` | Yes | Must contain `@`, valid format | `user@example.com` | empty, no `@`, `notanemail` |
| Password | `type="password"` | Yes | Min 8 chars (client-side check) | `password1` (8 chars) | 7 chars `passwor`, empty |

**Boundary values — Password:**
- 7 chars → "Password must be at least 8 characters" (client error)
- 8 chars → valid (boundary pass)
- 9 chars → valid

**Server-side validation (independent of client):**
- Duplicate email → HTTP 409 "An account with that email already exists"
- Invalid email (no `@`) → HTTP 400 "A valid email is required"
- Password < 8 → HTTP 400 "Password must be at least 8 characters"

**Client-side behaviour:**
- Empty email/password → browser native `required` validation (input focuses, no submit)
- Invalid email format → browser native email validation
- Password < 8 → JS check fires before API call → inline error paragraph

---

### 4b. Login Form (`/login`)

| Field | Type | Required | Constraint | Valid Partition | Invalid Partitions |
|-------|------|----------|-----------|----------------|-------------------|
| Email | `type="email"` | Yes | Valid format | Registered email | Empty, wrong format, unregistered |
| Password | `type="password"` | Yes | Any non-empty string | Correct password | Empty, wrong password |

**Client-side:** Browser `required` + `type="email"` native validation only (no custom JS check).

**Server-side:**
- Missing email → 400 "Email is required"
- Missing password → 400 "Password is required"
- Unknown email → 401 "Invalid email or password"
- Wrong password → 401 "Invalid email or password" *(same message — no user enumeration)*
- Error displayed in red banner below password field

---

### 4c. Add Book Modal

| Field | Type | Required | Constraint | Valid | Invalid |
|-------|------|----------|-----------|-------|---------|
| Title | text | Yes | Non-empty after trim | `"Dune"` | Empty, whitespace only |
| Author | text | Yes | Non-empty after trim | `"Frank Herbert"` | Empty, whitespace only |
| Genre | text | No | Optional | Any string, empty | — |
| Pages | number | No | `min="1"` (HTML attr); server accepts any number | `412`, `1` | `0`, negative, non-numeric |
| Cover URL | url | No | Optional; `type="url"` (no format validation client-side) | Valid URL, empty | Arbitrary text accepted by server |

**Client validation errors (shown as red text below field):**
- Title empty → "Title is required"
- Author empty → "Author is required"

**Server errors (currently no server response surfaced to UI on error):**
- Title/Author missing → 400 "Validation failed"

**Find Cover button:**
- Disabled when Title is empty
- Enabled as soon as Title has any value

---

### 4d. Add Review Modal

| Field | Type | Required | Constraint | Valid | Invalid |
|-------|------|----------|-----------|-------|---------|
| Book | select (combobox) | Yes | Must select a book | Any book from dropdown | Default "Select a book…" |
| Review Title | text | Yes | Non-empty after trim | Any non-blank string | Empty, whitespace only |
| Review | textarea | Yes | Non-empty after trim | Any non-blank string | Empty, whitespace only |

**Client validation errors:**
- Book not selected → "Please select a book"
- Review Title empty → "Title is required"
- Review empty → "Review is required"

**Note:** Dropdown only shows user's own books. If library is empty, no options appear (besides placeholder). Adding a review with an empty library is only possible via direct API call.

### 4e. Edit Review Modal

| Field | Type | Required | Behaviour |
|-------|------|----------|-----------|
| Book | read-only display | — | Shown as static text, cannot be changed |
| Review Title | text | Yes | Pre-populated with existing value |
| Review | textarea | Yes | Pre-populated with existing value |

**Client validation:** Same as Add Review (Title required, Review required).

## 5. API Endpoints Reference

| Method | Endpoint | Auth | Success | Error cases |
|--------|----------|------|---------|-------------|
| POST | `/api/auth/register` | No | 201 Created | 400 (invalid email/short password), 409 (duplicate email) |
| POST | `/api/auth/login` | No | 200 OK | 400 (missing fields), 401 (wrong credentials) |
| POST | `/api/auth/logout` | No | 200 OK | 500 (session destroy failure) |
| GET | `/api/auth/me` | Session | 200 OK | 401 (not authenticated) |
| GET | `/api/books` | Yes | 200 OK | 401 |
| GET | `/api/books/:id` | Yes | 200 OK | 401, 404 |
| POST | `/api/books` | Yes | 201 Created | 400 (missing title/author), 401 |
| PATCH | `/api/books/:id` | Yes | 200 OK | 400 (no valid fields), 401, 404 |
| DELETE | `/api/books/:id` | Yes | 204 No Content | 401, 404 |
| GET | `/api/reviews` | Yes | 200 OK (with joined bookTitle/bookAuthor) | 401 |
| POST | `/api/reviews` | Yes | 201 Created | 400 (missing fields), 401, 404 (book not found) |
| PATCH | `/api/reviews/:id` | Yes | 200 OK | 401, 404 |
| DELETE | `/api/reviews/:id` | Yes | 204 No Content | 401, 404 |

**Auth mechanism:** Session cookie (`connect.sid`). Credentials = `true` on CORS.

**Book status values (enum):** `unread` | `read` | `want-to-read`

**Cross-user isolation:** All book and review queries are scoped to `req.user.id` (from session). A user cannot access another user's data even by guessing IDs.

## 6. Risk Rankings

Ordered by **likelihood of bugs × business impact**:

| Rank | Feature | Risk | Reasoning |
|------|---------|------|-----------|
| 1 | **Authentication (login/register/session)** | Critical | Gate to entire app. Session bugs = locked-out users or data leakage. Duplicate email handling, password length edge cases. |
| 2 | **Add/Update/Delete Book (Library CRUD)** | Critical | Core feature. Status toggle logic has 3-way state machine (`unread` ↔ `read` ↔ `want-to-read`) — easy to get wrong. Delete is irreversible. |
| 3 | **Book Status Transitions** | Critical | Status drives both Library and Reading List views. A bad toggle leaves books in wrong state. "Want to Read" and "Remove" are contextual button labels — fragile to label changes. |
| 4 | **Add/Edit/Delete Review** | High | New feature (recently added). Full CRUD + modal lifecycle. Book dropdown populated from library — empty library edge case. |
| 5 | **Reading List (Derived View)** | High | Filtered view derived from book store. No separate fetch. Risk: state sync issues if books update while on reading list. |
| 6 | **Form Validation (client + server)** | High | Two validation layers (client JS + server). Divergence between them is a common bug source. Password boundary (7 vs 8 chars) is testable boundary value. |
| 7 | **Auth Protection / Route Guards** | High | ProtectedRoute redirects, reverse guard (auth → library). Session expiry behaviour untested. |
| 8 | **Find Cover (External API)** | Medium | Calls Open Library (third-party). Network errors, missing covers, and button enable/disable state are testable. Should be mocked in tests. |
| 9 | **Empty States** | Medium | Each page has a distinct empty state + CTA. CTAs navigate to other pages — testable user flow. |
| 10 | **Error States** | Medium | Loading/error UI in ReviewsPage. API failure handling in Library and Reading List pages. |
| 11 | **404 Page** | Low | Simple. "Back to My Library" link navigates to `/library`. |
| 12 | **Navigation / Layout** | Low | Static sidebar + navbar. Low logic, but visual regressions likely here. |

## 8. Accessibility Test Candidates

| Page | Priority | Notes |
|------|----------|-------|
| `/login` | High | Landmark structure, form labels, error announcement |
| `/register` | High | Same as login; "Minimum 8 characters" hint association |
| `/library` | High | Complex page: button grid on each card; focus management when modal opens/closes |
| `/reviews` | High | ReviewCard action buttons need accessible labels; modal focus trap |
| AddBookModal | High | Dialog role, focus trap, `aria-modal`, `aria-labelledby`, Escape key |
| AddReviewModal | High | Same modal concerns; combobox label association |
| EditReviewModal | High | Same modal concerns; read-only Book field accessibility |
| `/reading-list` | Medium | Same card structure as Library |
| 404 page | Low | Minimal content; heading and link |

**Known a11y observations from MCP snapshots:**
- All dialogs use `role="dialog"`, `aria-modal="true"`, `aria-labelledby` — correct.
- Close buttons use `aria-label="Close"` — correct.
- Form inputs use `<label for="...">` associations — correct.
- BookCard action buttons use `title` attribute for accessible name (e.g. "Delete book") — should verify with axe.
- Status badge uses generic `<div>` — may need `aria-label` or `role` for screen readers.
